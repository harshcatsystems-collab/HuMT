1. The 60-Second Pitch

The Shopify Story

Tobias Lütke (Shopify CEO) ran autoresearch on their Liquid templating engine overnight. 93 automated commits. 53% faster parsing. 61% fewer allocations.

Cost: ~$100 in API calls. Time: While he slept.

Autoresearch is a workflow pattern — not a product — for closed-loop autonomous experimentation. An AI agent reads a goal, proposes changes, runs experiments, keeps what works, discards what doesn't. Repeat overnight.

The math:

  • Manual A/B testing: ~30 experiments/year
  • Autoresearch: ~150 experiments/night → 36,500/year
The Punchline

If you can define "good" as a number, you can autoresearch it.

2. Is My Problem a Good Fit?

Answer three questions. Takes 5 minutes.

Question 1: Do I have a metric that's automatically computable?

No human judgment needed to score. A script can calculate it.

✅ YES examples:
  • Chat depth per session (count messages)
  • Push notification open rate (events / sends)
  • CTR on homepage rail (clicks / impressions)
  • IR × TCR composite (math on existing metrics)
❌ NO examples:
  • "Content quality" (requires human review)
  • "User satisfaction" (unless you have a proxy)
  • "Brand alignment" (subjective)

Fix: Define a proxy metric. "Content quality" → first-episode completion rate.

Question 2: Do I have ONE thing I can change?

A prompt, a config file, a ranking weight, a copy variant. Single, bounded scope.

✅ YES examples:
  • Character persona config YAML
  • Push notification copy template
  • Homepage rail ordering weights
  • Promo title/thumbnail selection
❌ NO examples:
  • "The entire recommendation system"
  • "Our content strategy"
  • "The user experience"

Fix: Narrow scope. "Recommendation system" → "rail 3 ranking weights."

Question 3: Do I have enough volume?

>100 events/day for the metric. Enough to detect signal.

✅ YES examples:
  • 1000+ chat sessions/day
  • 10K+ push notifications/day
  • 100K+ homepage impressions/day
❌ NO examples:
  • New feature with 50 users/week
  • B2B sales (10 deals/month)

Fix: Use synthetic testing first, or wait for volume.

All Three YES?

You're ready. Proceed to Section 3.

Any NO?

Fix that constraint first. The playbook tells you how above.

3. Writing Your Eval (The Hard Part)

This is where most people fail. The eval is not a test suite — the eval IS the product spec.

For deeper guidance on eval design, see Aakash Gupta's PM guide.

📋 Template: The Eval Spec
name: [your_eval_name]
surface: [what you're optimizing]

primary_metric: [the number]
calculation: [exact formula]

guardrails:
  - hard_reject_if: [condition 1]
  - hard_reject_if: [condition 2]
  - hard_reject_if: [condition 3]

success_threshold: [% improvement to count as winner]
volume_needed: [events for statistical significance]
proxy_correlation: [what true metric this should predict]
Example: Character Board Chat Depth
name: character_chat_depth_v1
surface: Character board conversations

primary_metric: weighted_chat_depth
calculation: messages_per_session × quality_multiplier

guardrails:
  - hard_reject_if: avg_user_message_length < 8 chars
  - hard_reject_if: bot_message_length > 3× user_message_length
  - hard_reject_if: user_sentiment_score < 0.3
  - hard_reject_if: repetition_ratio > 40%

success_threshold: 5% improvement
volume_needed: 200 conversations per variant
proxy_correlation: 7-day return rate (target r > 0.3)
Example: Push Notification Effectiveness
name: push_notification_v1
surface: Push notification copy + timing

primary_metric: open_rate × D1_return_rate
calculation: (opens / sends) × (D1_actives / openers)

guardrails:
  - hard_reject_if: unsubscribe_rate > 2× baseline
  - hard_reject_if: open_rate < 50% of baseline
  - hard_reject_if: D1_return < baseline

success_threshold: 10% improvement
volume_needed: 5000 sends per variant
proxy_correlation: D7 retention (target r > 0.4)

Common Eval Mistakes

Mistake Problem Fix
"Improve engagement" Not a number Define: sessions × avg_duration × return_rate
Raw message count Can be gamed (short messages) Add quality multiplier + length guardrail
5% threshold Too tight — noise looks like signal Use 10-15% for early experiments
No guardrails Agent will game the metric Always add 3+ hard reject conditions
No proxy check Optimizing wrong thing Track correlation with true business metric

4. Writing Your Goal Spec (program.md)

The goal spec tells the agent what to optimize and how. This is adapted from Karpathy's original format.

📋 Template: Goal Spec
# Goal
[One sentence: what you're optimizing for]

# The Asset
File: [path to the config/prompt/template]
Format: [YAML/JSON/text]

# What You Can Change (Mutable)
- [field 1]: bounds [min-max] or options [a, b, c]
- [field 2]: bounds [min-max]
- [field 3]: free text, max [N] chars

# What You Cannot Change (Frozen)
- [field A]: [reason]
- [field B]: [reason]
- Safety guardrails: NEVER modify

# The Loop
1. Read current config and last 5 experiment results
2. Propose ONE change with hypothesis (why you think this helps)
3. Run eval (calls eval harness, waits for score)
4. If score > baseline + [threshold]%, commit change
5. If score <= baseline, revert and try different approach
6. Log result with hypothesis + outcome
7. NEVER STOP until [stopping condition]

# Stopping Conditions
- Stop after [N] experiments
- Stop if [M] consecutive failures
- Stop at [time]

# Constraints
- Max [X] changes per experiment (usually 1)
- Mutation bounds: max ±[Y]% per field per experiment
- Must maintain [invariant]
Example: Character Board Goal Spec
# Goal
Maximize weighted chat depth for Haryanvi action characters

# The Asset
File: configs/characters/haryanvi_action_hero.yaml
Format: YAML with immutable/semi-mutable/mutable sections

# What You Can Change (Mutable)
- dialect_density: bounds [0.3, 0.9]
- question_frequency: bounds [0.1, 0.5]
- emoji_usage: bounds [0, 0.3]
- message_length_target: bounds [20, 150] chars
- cliffhanger_frequency: bounds [0, 0.4]
- memory_callback_rate: bounds [0.1, 0.5]

# What You Cannot Change (Frozen)
- character_name: Core identity
- backstory: Canonical lore
- safety_guardrails: Content compliance
- content_rating: Legal requirement

# The Loop
1. Read current config and last 10 experiment results
2. Propose ONE field change with hypothesis
3. Run 50 synthetic conversations via eval harness
4. If weighted_chat_depth > baseline + 5%, commit
5. If not, revert and try different field
6. Log: field, old_value, new_value, hypothesis, score, outcome
7. NEVER STOP until 150 experiments or 8 hours

# Stopping Conditions
- Stop after 150 experiments
- Stop if 20 consecutive failures
- Stop at 7:00 AM IST

# Constraints
- Max 1 field change per experiment
- Mutation bounds: max ±0.15 per field
- Must pass all 4 guardrails in eval spec

5. The Loop Options (Pick Your Speed)

Option Speed User Risk When to Use
Pure Synthetic ~150/night Zero Early exploration, high uncertainty, character boards
Shadow Testing ~10/day Near-zero Validating synthetic winners before live
Live Bandit ~6/day Low (with guardrails) High-traffic surfaces, push notifications
Hybrid 150 + 6 Optimal Best of both — recommended for most surfaces

The Hybrid Loop (Recommended)

The 24-Hour Clock

Time Phase What Happens
11pm-7am Synthetic Screening Agent runs ~150 mutations. Keeps anything >5% better. Cost: ~$80.
7am-9am Agent Curates Reviews ~15 winners. Groups by variable. Max 2 per type.
9am-9pm Live Bandit Thompson Sampling. 5-10% per variant. Auto-kills losers.
9pm-11pm Promotion Winners → new baseline. Agent logs learnings.

The Narrowing Funnel:

~150 mutations → ~15 pass synthetic → 6-8 go live → 1-2 winners → new baseline → repeat

7. FAQ & Troubleshooting

"My synthetic winners don't transfer to real users"

Problem: Your synthetic personas don't match real users.

Fix: Build personas from real chat logs. Update them weekly based on live data. Track synthetic-to-real correlation and tune until >70%.

"The agent keeps trying the same thing"

Problem: Goal spec is too narrow, or agent isn't learning from failures.

Fix: Expand the mutable field list. Add explicit instruction: "If last 3 experiments on [field] failed, try a different field."

"Metrics improved but business outcomes didn't"

Problem: Your eval is optimizing a bad proxy.

Fix: Check proxy-true correlation. If chat depth doesn't correlate with 7-day return (r < 0.3), your eval is wrong. Redesign it.

"I don't have engineering resources"

Fix: Start manual. Run the loop by hand for 1 week:

  1. Morning: Write 3 variants of your asset
  2. Day: Split traffic manually (GrowthBook)
  3. Evening: Check metrics, pick winner
  4. Next day: Repeat with winner as baseline

This proves the pattern works. Then automate.

"How do I know if I'm reward-hacking?"

Warning signs:

  • Metric improves but user complaints increase
  • Metric improves but downstream conversion drops
  • Variants look "weird" to humans
  • Agent finds one trick and keeps exploiting it

Fix: Add guardrails. Run constitutional checks. Do blind human review every 5th generation.
