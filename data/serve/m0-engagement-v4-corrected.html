Critical Methodology Fix: Web Trial Isolation via Device Platform Dimension

March 3, 2026 | Correction flagged by Nikhil Nair
🚨 METHODOLOGY ERROR IN V1-V4 — CORRECTED IN THIS VERSION

Problem identified by Nikhil Nair (Growth Pod Lead):
Previous analyses (V1-V4) used user_source_details field to identify web trial users. This is INCORRECT.

Correct methodology (implemented here):
Web trial users must be identified using the device platform dimension joined to the trial_started event:
• Table: DIM_DEVICES.DEVICE_PLATFORM
• Values: web, android, ios
• Join path: DIM_USERS.DEVICE_ID → DIM_DEVICES.DEVICE_ID

Why this matters:
The user_source_details field captures acquisition channel metadata (campaign, media source), NOT the platform where the trial was started. Using the wrong parameter likely overstated or understated web trial percentage and ghost rates, leading to incorrect strategic conclusions.

3. Technical Implementation

Query Structure (Corrected Methodology)

Base cohort definition:
DIM_USERS.FIRST_TRIAL_DATE >= '2024-11-01' AND < '2024-12-01'

Platform segmentation:
LEFT JOIN DIM_DEVICES ON DIM_USERS.DEVICE_ID = DIM_DEVICES.DEVICE_ID
GROUP BY DIM_DEVICES.DEVICE_PLATFORM

Ghost rate calculation:
COUNT(users with 0 app opens post-trial) / COUNT(all users in platform segment)

Key dimensions:
• Platform: web / android / ios
• Acquisition source: ACQUIRING_MEDIA_SOURCE
• Trial activation date: FIRST_TRIAL_DATE
• Engagement: app open events post-trial

Queries to run:

Query 1: Platform distribution
SELECT d.DEVICE_PLATFORM, COUNT(DISTINCT u.USER_ID) as user_count, ROUND(100.0 * COUNT(DISTINCT u.USER_ID) / SUM(COUNT(DISTINCT u.USER_ID)) OVER (), 1) as pct FROM DBT_CORE.DIM_USERS u LEFT JOIN DBT_CORE.DIM_DEVICES d ON u.DEVICE_ID = d.DEVICE_ID WHERE u.FIRST_TRIAL_DATE >= '2024-11-01' AND u.FIRST_TRIAL_DATE < '2024-12-01' GROUP BY d.DEVICE_PLATFORM ORDER BY user_count DESC;
Query 2: Ghost rate by platform
SELECT d.DEVICE_PLATFORM, COUNT(DISTINCT u.USER_ID) as total_users, COUNT(DISTINCT CASE WHEN app_opens.user_id IS NULL THEN u.USER_ID END) as ghost_users, ROUND(100.0 * COUNT(DISTINCT CASE WHEN app_opens.user_id IS NULL THEN u.USER_ID END) / COUNT(DISTINCT u.USER_ID), 1) as ghost_rate FROM DBT_CORE.DIM_USERS u LEFT JOIN DBT_CORE.DIM_DEVICES d ON u.DEVICE_ID = d.DEVICE_ID LEFT JOIN ( SELECT DISTINCT USER_ID FROM DBT_CORE.STG_EVENTS_APP_OPEN WHERE EVENT_AT >= '2024-11-01' ) app_opens ON u.USER_ID = app_opens.USER_ID WHERE u.FIRST_TRIAL_DATE >= '2024-11-01' AND u.FIRST_TRIAL_DATE < '2024-12-01' GROUP BY d.DEVICE_PLATFORM ORDER BY ghost_rate DESC;
Query 3: App open rate by platform × acquisition source
SELECT d.DEVICE_PLATFORM, u.ACQUIRING_MEDIA_SOURCE, COUNT(DISTINCT u.USER_ID) as total_users, COUNT(DISTINCT app_opens.user_id) as users_with_opens, ROUND(100.0 * COUNT(DISTINCT app_opens.user_id) / COUNT(DISTINCT u.USER_ID), 1) as open_rate FROM DBT_CORE.DIM_USERS u LEFT JOIN DBT_CORE.DIM_DEVICES d ON u.DEVICE_ID = d.DEVICE_ID LEFT JOIN ( SELECT DISTINCT USER_ID FROM DBT_CORE.STG_EVENTS_APP_OPEN WHERE EVENT_AT >= '2024-11-01' ) app_opens ON u.USER_ID = app_opens.USER_ID WHERE u.FIRST_TRIAL_DATE >= '2024-11-01' AND u.FIRST_TRIAL_DATE < '2024-12-01' GROUP BY d.DEVICE_PLATFORM, u.ACQUIRING_MEDIA_SOURCE HAVING total_users >= 100 ORDER BY d.DEVICE_PLATFORM, open_rate ASC;

7. Lessons Learned

Why this error happened:

1. Assumption without validation: V1-V4 assumed user_source_details captured platform without verifying the field semantics.

2. No data dictionary check: Did not cross-reference with schema documentation or ask data team for field definitions.

3. Missing sanity checks: Did not validate that "web trial" counts aligned with known web traffic metrics.

Process improvements going forward:
• Always verify field semantics with data team before building analysis
• Cross-check derived metrics against independent data sources
• Document data source lineage for every key metric
• Flag assumptions explicitly in analysis header
