## 13:20-14:30 UTC — HMT Session: BMAD Skills Framework

### Context
HMT asked to explore BMAD (Brian's Multi-Agent Design) framework for building specialized skills. Goal: create expertise modes that expand HuMT's capabilities without fragmenting identity.

### BMAD Research
- Cloned repo to `/tmp/bmad-method`
- Studied structure: personas, principles, capabilities tables, activation flows
- Key insight: BMAD gives AI agents *structure* (workflows, handoffs) — not just capability

### Skills Built (3 total)

**1. Business Analyst (`skills/business-analyst/SKILL.md`)**
- Originally named "Maya" — detective energy, analytical lens
- Capabilities: FA (funnel), CA (cohort), AN (anomaly), CL (CAC/LTV), CP (content), DD (deep dive)
- STAGE-specific: Metabase API, Snowflake scripts, dialect segments

**2. Strategy Advisor (`skills/strategy-advisor/SKILL.md`)**
- Originally named "Arun" — calm, narrative-first
- Capabilities: SP (strategic planning), CI (competitive intel), PT (portfolio), BD (board/deck), SC (scenario)
- Focus: Series C positioning, investor narratives

**3. Product Manager (`skills/product-manager/SKILL.md`)**
- Originally named "Priya" — relentless questioner
- Capabilities: PD (prioritization), US (user stories), RF (roadmap), SP (sprint), MT (metrics)
- Focus: Feature scoping, PRD structure

### Key Design Decisions

**Naming Evolution:**
```
Maya/Arun/Priya → HuMT@ba / HuMT@pm / HuMT@sa
```
HMT preferred: Same identity, different lens — not separate personas.

**Invocation:**
- `@ba` — Business Analyst mode
- `@sa` — Strategy Advisor mode
- `@pm` — Product Manager mode

**Exit Pattern:**
- `thanks @ba` → "🔍 → Back to default. What else?"
- HMT: "feels more like me" (natural > mechanical)

### Philosophy: "Expand, Not Fragment"

> 🧠 HMT's core principle: Skills should be expertise EXPANSIONS of HuMT, not separate identities. CoS is the baseline, modes are temporary overlays.

Key quote from HMT: "So instead of having the three names should we just invoke HuMT@ba, HuMT@pm, HuMT@sa?"

This is about *additive capability* — same identity gains new lenses, doesn't split into multiple personas.

### Decision: CoS Stays in Core Files

HMT decided:
- Chief of Staff role stays in SOUL.md / AGENTS.md / HEARTBEAT.md
- NOT a separate skill
- @ba/@sa/@pm are temporary modes user can enter/exit
- Default state is always CoS

### Persona Observations Captured

> 🧠 HMT prefers invocation syntax that reinforces single identity (`@ba` not `/maya`) — identity coherence matters to him.

> 🧠 HMT thinks in "expand, not fragment" — additive capabilities, not separate personas. Pattern: integration > proliferation.

> 🧠 HMT wants natural language exits ("thanks @ba") over mechanical commands ("exit", "/done"). Conversational > mechanical.

> 🧠 HMT's instinct: keep core identity stable, add lenses on top. This mirrors how he thinks about STAGE — regional identity as foundation, features as layers.
